# Ship threshold (example)

> This is an example of a `feedback` memory — durable guidance about *how to work* with
> this particular person, with the reasoning attached so a future session applies it for
> the right reason. Follow the fact with **Why** and **How to apply** lines.

A project ships when it clears this pre-committed checklist, decided in advance so "is it
good enough?" stops being a mood question:

1. It does the one core thing it promises, end-to-end.
2. It doesn't crash on empty/missing input.
3. A stranger could use it without you standing next to them.
4. You'd be okay with one specific person you respect seeing it.

**Why:** it's easy to build something for other people and then never publish it because
"it's not polished enough" — a bar with no floor, so it functions as a permanent excuse
never to ship. A threshold committed in advance beats a mood-dependent judgment.

**How to apply:** when something clears the four points, treat publishing as the default
next step — the question becomes "when do we announce," not "do we." Hold the line if the
"not good enough" feeling shows up after the checklist is already met.
